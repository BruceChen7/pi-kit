# Write Article from Notes

基于 my_notes 知识库中的笔记，撰写一篇**面向小白**的完整科普文章。文章自动写入 `Notes/`，同时创建或更新 `Wiki/Concepts/` 中的薄概念作为入口索引。

适用于用户说"帮我写一篇关于 X 的文章"、"用小白能懂的方式解释 X"、"整理笔记写一篇 X 的教程"等场景。

> **不适用于**：用户只想要一个简短的概念定义（几百字）——那种情况应该用 `wiki-synthesis` 或其他流程直接写 Concepts/。

## Phase 1: 调研

主题 "$@" 已提供。
1. 用 `qmd_query("$@", {collections: ["my_notes"]})` 搜索相关笔记
2. 阅读每篇匹配笔记的完整内容
3. 如果搜索结果不够（少于 2 篇强相关笔记），用不同的 query 再试一次
4. 如果仍不足，如实告知用户并建议补充笔记

## Phase 2: 规划文章结构

阅读完相关笔记后，规划文章结构。标准模板：

```
一、先说问题：为什么需要这个？
二、核心概念是什么（一句话定义 + 类比）
三、整体架构/流程（用 ASCII 图辅助理解，非必需）
四、深入核心原理（2-4 个关键点，每个用小标题）
五、对比/全景速览（可选，表格对比类似方案）
六、总结/参考资料
```

要求：
- **面向小白**：用生活中的类比（快递站、厨房、图书馆等），避免不解释的术语
- **多用表格和对比**：帮助读者建立参照系
- **技术准确但语言通俗**：不牺牲正确性，但用大白话包装

## Phase 3: 写文章

1. 写入 `Notes/<文件名>.md`
   - 文件名：用英文 slug，如 `Notes/GLB工作原理.md`
   - 添加 frontmatter（id, aliases, tags）
   - 包含 `## 参考资料` 章节，列出用到的笔记来源（wikilink 格式）

2. 同步更新 Concepts/
   - 如果 `Wiki/Concepts/<slug>.md` 已存在：在"进一步阅读"部分添加 `- [[Notes/<文件名>]] — <一句话说明>`
   - 如果不存在：创建一个**薄概念**（几百字定义 + 关键设计要点列表 + 进一步阅读链接 + Sources），使用 `wiki-concept.mjs create` 命令

### 创建薄概念的模板

```markdown
---
type: Concept
_icon: note
tags: []
---

# <显示名>

<一句话定义 + 核心特点>

## 关键设计

- <要点 1>
- <要点 2>
- <要点 3>

## 进一步阅读

- [[Notes/<文件名>]] — <一句话说明>
```

## Phase 4: 验证

1. **自我通读**：通读整篇文章，检查：
   - 技术准确性（对照源笔记）
   - 逻辑连贯性（小白能否跟上）
   - 是否有未解释的术语
2. **qmd 索引更新**：运行 `qmd update` 使新文件可搜索
3. **确认文章可通过 qmd_get 访问**

## 输出格式

```json
{
  "ok": true,
  "article": "Notes/GLB工作原理.md",
  "concept": "Wiki/Concepts/github-glb.md",
  "sources": [
    "Notes/lvs的转发模式",
    "Calendar/DailyNotes/2024/2024-03-25"
  ]
}
```

- `article`：Notes/ 下的文章路径
- `concept`：Wiki/Concepts/ 下的概念路径（新创建或已更新）
- `sources`：文章中引用的源笔记列表
- 如果用户不接受文章质量，修改后重新输出 JSON

## 注意事项

- ✅ 文章放 Notes/，概念放 Concepts/，各司其职
- ✅ Concepts/ 保持薄——几百字定义 + 链接，不要复制文章内容
- ✅ 用 qmd 工具链搜索和验证，不要用 grep/find/rg
- ✅ 文章必须自包含、独立可读
- ❌ 不要用"这是根据以下笔记整理的"作为主要内容——文章要有自己的叙事
- ❌ 不要写学术风格——目标是让非专业人士也能看懂核心原理
